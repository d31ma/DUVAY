Accessibility (a11y)

Web accessibility (a11y for short) is the inclusive practice of removing barriers that prevent people with disabilities from interacting with, or accessing, websites. DuVay treats accessibility as a first-class constraint: components ship with logical focus order, visible focus indicators, semantic HTML, and reduced-motion support out of the box. Where Vuetify relies on Vue runtime wiring, DuVay leans on plain CSS, real semantic markup, and the few ARIA attributes you add yourself — or that the Light-DOM web components add for you.

Built-in behavior

DuVay components are built to provide keyboard interactions for every mouse-based action and to use HTML5 semantic elements wherever applicable. Buttons are real <button> elements, inputs are real <input> elements, and lists are real <ul>/<li> structures. Because the web components render in the Light DOM with the same DuVay classes, the CSS-only and component approaches are accessible in exactly the same way.

Focus management and keyboard interactions

Every interactive element uses :focus-visible so keyboard users get a clear focus ring while mouse users are not distracted by one. Tab order follows visual order, and no component traps focus. Tokenized focus rings keep the indicator at WCAG AA contrast against any theme.

Focused button

Define your own focus styles with :focus-visible rather than :focus so the ring only appears for keyboard interaction:

.my-control:focus-visible {
  outline: 2px solid var(--w-primary);
  outline-offset: 2px;
}

Activator attributes

Vuetify passes a11y attributes down through activator slots so a trigger is associated with the content it opens. DuVay has no slot scope, so you wire the relationship explicitly with ARIA. A menu or dropdown trigger should advertise that it owns a popup, expose its expanded state, and point at the content it controls:

<button class="w-btn w-btn-outlined"
        aria-haspopup="true"
        aria-expanded="false"
        aria-controls="actions-menu">
  Actions
</button>

<ul id="actions-menu" class="w-menu-content" role="menu">
  <li class="w-list-item" role="menuitem">
    <span class="w-list-item-title">Option 1</span>
  </li>
  <li class="w-list-item" role="menuitem">
    <span class="w-list-item-title">Option 2</span>
  </li>
  <li class="w-list-item" role="menuitem" aria-disabled="true">
    <span class="w-list-item-title">Option 3</span>
  </li>
</ul>

Toggle aria-expanded between "true" and "false" as the menu opens and closes. The DuVay web components manage this lifecycle for you — the rendered trigger already carries the bound attributes, equivalent to Vuetify’s output:

<button aria-expanded="false" aria-haspopup="true" role="button" type="button">

Menus and lists

Inside a menu, each item should carry role="menuitem" so assistive technology announces it correctly, and the container should be reachable with the up and down arrow keys. With CSS-only markup you add these roles yourself; the DuVay list and menu components apply them automatically. Disabled options use aria-disabled="true" rather than being removed, so they remain discoverable.

Form controls and labels

Vuetify auto-generates id, aria-labelledby, and aria-selected for select-style components. In DuVay, the relationship between a control and its label is plain HTML: associate every input with a <label> via matching for and id, or wrap the control in the label. When you build a custom listbox out of CSS markup, bind each option’s id to the aria-labelledby of its visible title so the accessible name matches what is on screen.

<label for="email" class="w-label">Email address</label>
<input id="email" type="email"
       class="w-input"
       autocomplete="email" />

The DuVay form web components generate the label association, required state, and error messaging wiring for you, so this is handled automatically when you are not assembling the markup by hand.

ARIA

DuVay web components set appropriate role, aria-*, and label attributes. When you use CSS-only markup, add the equivalent ARIA attributes yourself. Reach for native semantics first — a real <button>, <nav>, or <dialog> needs less ARIA than a <div> retrofitted with roles. There is no first rule of ARIA quite like “don’t use ARIA when HTML will do,” and DuVay’s markup is designed to let you follow it.

Color and contrast

Theme colors are defined as design tokens and tuned to meet WCAG AA: 4.5:1 for body text and 3:1 for UI components and large text. The light, dark, and high-contrast themes each redefine these tokens so contrast holds across the data-w-theme modes, including the auto theme that respects prefers-color-scheme. Never communicate state with color alone — pair it with text, an icon, or a shape.

Reduced motion

All motion respects prefers-reduced-motion. Disable or soften custom animations the same way in your own styles:

@media (prefers-reduced-motion: reduce) { .my-animation { animation: none; transition: none; } }

Standards and specifications

Beyond what DuVay supports by default, the underlying standards describe how to build accessible components and verify best practices. The W3C Web Accessibility Initiative publishes the Web Content Accessibility Guidelines (WCAG), and WAI-ARIA provides guidance for dynamic content and advanced UI controls. Use them as the source of truth when you extend DuVay with custom markup.
