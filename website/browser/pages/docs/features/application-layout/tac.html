Application layout

DuVay provides the building blocks for common application shells — app bars, navigation drawers, system bars, footers, bottom navigation, and a main content area. The layout works on an outside-in principle: each shell component reserves space along one edge (top, bottom, left, or right) using sticky/fixed positioning, and w-main flows into whatever space is left. Unlike Vuetify, there is no JavaScript layout engine — offsets are coordinated entirely with CSS positioning, position, and inset.

The layout system

A shell is assembled from a small set of cooperating components. Each one claims an edge; w-main occupies the remaining free space.

ComponentReservesDescription
w-app-bartop / bottomContainer for navigation, branding, search, and actions.
w-system-bartopA thin status strip that replaces the native device system bar.
w-navigation-drawerleft / rightPermanent or temporary container for site navigation links.
w-footerbottomA replacement for the default HTML footer.
w-bottom-navigationbottomPersistent navigation, typically for smaller devices.
w-main—Holds page content and fills the remaining free space.

Note: the examples below use w-layout as the root rather than w-app, because w-app defaults to a minimum height of 100dvh (full viewport). In production, use w-app as the root of your application.

Usage

By default, shell components reserve space in the order they appear in the markup. This source order decides how edges are claimed when two components compete for the same corner. Compare an app bar placed before versus after the navigation drawer:

  • When w-app-bar comes before the drawer, the bar spans the full width of the screen and the drawer sits beneath it.
  • When the bar comes after the drawer, the drawer reaches the top edge and the bar only uses the leftover width beside it.
<!-- App bar first: bar spans full width -->
<div class="w-app">
  <header class="w-app-bar">…</header>
  <div class="w-layout">
    <aside class="w-navigation-drawer w-navigation-drawer--permanent">…</aside>
    <main class="w-main">…</main>
  </div>
</div>

<!-- Drawer first: drawer reaches the top edge -->
<div class="w-app">
  <div class="w-layout">
    <aside class="w-navigation-drawer w-navigation-drawer--permanent">…</aside>
    <div>
      <header class="w-app-bar">…</header>
      <main class="w-main">…</main>
    </div>
  </div>
</div>

Some components accept a location placement. A navigation drawer defaults to the left edge; add w-navigation-drawer--right (or location="right" on the web component) to anchor it on the right. You can place one drawer on each side.

Sizing shell components

Shell components accept an explicit size so they reserve a fixed band of space. Set a width on drawers and a height on bars and footers — with classes use an inline style custom property or a utility, and with web components use the width / height attributes. Multiple components of the same type may stack along the same edge to recreate richer shells (for example a chat app with a server rail, a channel list, and a content column).

<!-- Two stacked drawers form a wide sidebar region -->
<aside class="w-navigation-drawer w-navigation-drawer--permanent w-navigation-drawer--rail">…</aside>
<aside class="w-navigation-drawer w-navigation-drawer--permanent" style="width:240px;">…</aside>
<w-navigation-drawer permanent rail>…</w-navigation-drawer>
<w-navigation-drawer permanent width="240">…</w-navigation-drawer>

Complex layouts

Because each component simply claims an edge, you can compose dense application shells — for example the general layout of a chat app like Discord: a narrow server rail, a wider channel drawer, a top app bar over the content, and a member drawer on the right. Stack same-type components and give each a fixed size to recreate the structure.

<div class="w-app">
  <div class="w-layout">
    <!-- narrow server rail -->
    <aside class="w-navigation-drawer w-navigation-drawer--permanent w-navigation-drawer--rail">…</aside>
    <!-- channel list -->
    <aside class="w-navigation-drawer w-navigation-drawer--permanent" style="width:240px;">…</aside>
    <!-- content column with its own bar -->
    <div>
      <header class="w-app-bar"># general</header>
      <main class="w-main">…</main>
    </div>
    <!-- members on the right -->
    <aside class="w-navigation-drawer w-navigation-drawer--permanent w-navigation-drawer--right">…</aside>
  </div>
</div>

Dynamic layouts and order

Placing components in the correct markup order is enough for most layouts. When you need to override that order — for example to push the app bar above a drawer it appears after in the DOM — use CSS order on the flex layout container, or a layout utility. All shell members default to an order of 0; members with an equal order fall back to their markup sequence.

/* Force the app bar above the drawer regardless of markup position */
.w-app-bar { order: -1; }

The w-layout-item wrapper exists for this purpose: drop a shell element inside it and control placement and order through positioning utilities, keeping the rest of the shell declarative.

Coordinating offsets

Vuetify computes layout sizes with a JavaScript engine and a useLayout composable. DuVay has no such engine — instead, the w-layout container coordinates offsets purely with CSS. Each shell member uses sticky or fixed positioning and reserves space via position and inset; w-main reads those reserved bands through the flex/grid flow and the drawer width custom properties, so its content never slides beneath a bar or drawer.

If you need to read a shell member’s size or position at runtime (the equivalent of Vuetify’s getLayoutItem), query the element directly — there is no composable to inject:

const drawer = document.querySelector('.w-navigation-drawer');
const { width } = drawer.getBoundingClientRect();
document.documentElement.style.setProperty('--w-drawer-offset', width + 'px');

Because offsets are plain custom properties, you can override them in CSS or set them from script without coupling your components to a layout runtime.

Scrolling techniques

The shell stays put while content scrolls. Pin a bar with w-app-bar--sticky (or sticky on the web component) so it remains visible during scroll, and let the main region own its own scroll with w-main--scrollable. App bars also support scroll behaviors — hide, elevate, collapse, and fade-image — that respond as the user scrolls past a threshold.

<header class="w-app-bar w-app-bar--sticky">…</header>
<main class="w-main w-main--scrollable">…</main>
<w-app-bar sticky scroll-behavior="hide" scroll-threshold="48">…</w-app-bar>

Responsive shells

Combine layout classes with the grid system and utility classes to adapt the shell to any viewport — swap a permanent drawer for a temporary one (w-navigation-drawer--temporary) on small screens, or replace a side drawer with bottom navigation. Respect prefers-reduced-motion for any drawer or bar transitions.

Accessibility

Give the app bar a landmark role (it renders as <header>), use <nav> for drawer and bottom navigation contents, and pair the nav toggle (w-app-bar-nav-icon) with a clear aria-label. Temporary drawers must trap focus while open, restore focus on close, and be dismissible with Esc. Keep touch targets at least 44×44 px and ensure a visible :focus-visible state on every interactive shell control.
