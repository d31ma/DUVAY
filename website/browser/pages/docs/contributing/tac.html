Contributing

DuVay is an open framework and contributions are welcome. This page covers the repository layout, how to build and test locally, and the conventions every change must follow.

Repository layout

PathContents
src/*.cssDesign tokens, themes, reset, and component styles (plain CSS custom properties)
src/tokens.cssThe source of truth for colors, spacing, radii, shadows, and typography scales
src/themes.cssLight, dark, and high-contrast theme token values
src/components/*.cssPer-component styles, token-driven only
src/components/*.jsLight-DOM web components extending the base class
src/duvay.jsThe behavior layer for CSS-class usage (dropdowns, dialogs, toasts, theme cycling)
website/The documentation site (a Tac frontend app with file-system routing)

Building locally

Build the framework, then bundle the documentation site:

# Build the framework (CSS + JS bundles) bun run build # Bundle the documentation site bun run --cwd website bundle

Framework edits do not appear in the docs preview until both commands have run and the preview server has been restarted.

Running tests

Component behavior is covered by an automated suite:

bun run test:components

Run the tests before opening a pull request and make sure they pass.

Coding conventions

  • Design tokens only. Every color, spacing value, radius, shadow, and typography scale must reference a CSS custom property. No hard-coded hex values and no magic numbers.
  • Token naming. Follow the existing convention: --w-<category>-<name>, for example --w-primary, --w-space-4, --w-radius-lg.
  • Define every token in every theme. When you add a token, update the light, dark, and high-contrast theme files. Never leave a token undefined in an active theme.
  • Light DOM only. Web components must not use Shadow DOM. Generate the correct DuVay class names so static and component usage stay identical.
  • Accessibility is non-negotiable. Provide correct role and aria-* attributes, full keyboard navigation, visible :focus-visible states, and respect prefers-reduced-motion. Maintain WCAG AA contrast.
  • Stay lean. Every selector should earn its place. Prefer utility-class patterns over deeply nested selectors, and keep rem/em sizing (reserve px for hairline borders).
  • Keep docs in sync. Any change to a public API (new component, renamed attribute, removed event) must update the matching docs page in the same pull request.

Pull request checklist

  • bun run build completes without errors.
  • bun run --cwd website bundle builds the site with zero warnings.
  • bun run test:components passes.
  • New or changed tokens are documented with a comment block and a brief changelog entry.
  • Contrast ratios and responsive behavior were verified at key breakpoints.
